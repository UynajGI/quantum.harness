Before you tune a single setting, two choices set your ceiling: how you represent the circuit, and which software stack you run it on.

These two choices are general to quantum circuit simulation; the variational quantum eigensolver (VQE) — a parameterized circuit whose angles are tuned to minimize an energy — is the recurring example. Two high-level forks come first. Representation: contract the circuit as a tensor network, hold the full amplitude vector, or approximate it as a matrix-product state. Stack: which framework you build on. Get these right and the detailed tuning has room to work; get them wrong and no setting saves you.

This is the high-level map. Its companion — the expert interview — drills into the detailed decisions inside a chosen stack (precision, gradient method, contraction tuning, hardware split) and the rule an expert uses for each. 1. Which approach wins, when

Three ways to represent the same circuit state. The choice fixes the memory wall and whether the answer is exact or approximate — the first thing to settle, before any framework or setting.

Approach What it represents When it wins Works while Main limit
Tensor-network contraction Builds the circuit as a network of tensors and contracts only what the observable needs — never forms the full 2N amplitude vector if the path avoids it. The default for exact expectation values and gradients; pays off whenever a smart contraction order keeps the largest intermediate tensor small. the largest intermediate tensor still fits memory (≈ 32 × 16 on one GPU) The largest intermediate tensor — memory, not qubit count, is the wall.
Full statevector The full 2N complex amplitude vector, held in memory. Smaller systems, verification, sampling and noise studies, or when you genuinely need every amplitude. the full 2N state fits memory (≈ 33 qubits on an 80 GB GPU) Memory doubles with every added qubit; no structure exploited.
Matrix-product state (MPS) The state as a chain of small tensors linked by a bond dimension χ that caps how much entanglement it can carry. 1D or low-entanglement circuits with many qubits, where χ stays bounded. the result converges at a feasible bond dimension χ (100s of qubits if χ stays small) A truncation error that grows with entanglement; too-small χ silently biases the result.

For a deep, large variational circuit where you need gradients, tensor-network contraction is usually the right call — it is the regime the benchmark above lives in. Full statevector is the simplest and best for small systems and for anything needing the whole state. MPS is the lever for long, weakly-entangled circuits.

2. The software stack

The framework fixes what is easy. Two families: tensor-network simulators contract the circuit and scale with its structure; statevector simulators hold the full amplitude vector — simplest, and strongest at smaller qubit counts, sampling, and noise. The differentiator for a variational task is whether the gradient flows through the simulation efficiently.

Stack Family Backend / language Gradients GPU When it wins
TensorCircuit-NG tensor network Python on JAX / PyTorch / TensorFlow / NumPy reverse-mode automatic differentiation (AD) on JAX / TensorFlow / PyTorch; NumPy has none yes · multi-GPU / distributed Large, deep variational circuits that need gradients through the simulation at scale; cotengra path optimization is the speed lever. The focal tool of the interview.
quimb tensor network Python (autoray: NumPy / JAX / PyTorch / CuPy) via AD backend (JAX / PyTorch / TensorFlow / autograd) yes (CuPy / JAX) Flexible tensor-network research — general networks, MPS / PEPS, not just circuits. Shares the same cotengra contraction engine; lighter on circuit-VQE conveniences.
Qiskit Aer multi-method (statevector / density-matrix / MPS / stabilizer / TN) Python + C++ parameter-shift, finite-difference, and related methods via Qiskit Algorithms; no Aer-native AD yes (CUDA; cuStateVec / cuTensorNet) The IBM / Qiskit ecosystem; noise and density-matrix simulation, sampling, hardware-aligned workflows. Several methods (statevector, MPS, stabilizer) in one tool.
NVIDIA cuQuantum both (a backend SDK) C++ / Python (CUDA) low-level cuTensorNet gradient APIs; loop AD on caller / framework yes · native multi-GPU Maximum GPU / multi-GPU throughput as the engine under another framework (cuStateVec for statevector, cuTensorNet for contraction). No high-level VQE ergonomics of its own.
PennyLane + Lightning statevector (+ lightning.tensor MPS/TN) Python + C++ adjoint differentiation yes (lightning.gpu / kokkos) Differentiable quantum-machine-learning pipelines at moderate qubit counts; hardware-portable PennyLane workflows.
Yao.jl statevector-first (+ TN via YaoToEinsum) Julia built-in AD (ChainRules / Zygote) yes (CuYao / CUDA.jl) Julia-native workflows: fast statevector plus easy circuit manipulation for research prototyping.

When TensorCircuit-NG specifically wins: a large or deep variational circuit where you need the gradient through the whole simulation, fast and repeatedly — that is where end-to-end automatic differentiation plus just-in-time compilation plus a cotengra-optimized contraction path compound into the benchmark's single-GPU numbers. When a statevector tool is the better call: smaller qubit counts where the full state fits, or when you need sampling, realistic noise, or tight integration with a particular hardware ecosystem.

3. 2026 feasibility anchors

What this kind of run reaches by hardware regime. For tensor-network contraction the ceiling is the largest intermediate tensor, not the qubit count alone — read it off the contraction cost before committing (a detail the interview covers).

Laptop CPU
≲ 20–24 qubits, shallow
For correctness and debugging, not speed. Automatic differentiation needs a real backend (JAX / PyTorch), not plain NumPy — switch before timing anything.
Single GPU (H200-class), exact contraction
32 q × 16 layers — 17.86 s / eval
The workhorse regime. One warm energy-and-gradient evaluation of the benchmark VQE in 17.86 s, after a one-off contraction-path search and compile.
Single GPU + slicing
fits any contraction, slower
When the largest tensor won't fit, slicing fixes some indices and sums the pieces, capping memory in exchange for repeated work — exact, just slower.
Multi-GPU / distributed
scales with device count
Data-parallel for many independent terms or restarts; model-parallel (a distributed contractor) when a single contraction exceeds one device's memory.
